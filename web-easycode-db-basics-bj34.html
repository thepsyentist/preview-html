한 줄

"DB" 라고 부르는 단어 하나에 사실 3개의 다른 것 이 섞여 있다 — Database / DBMS / Database System. 이걸 구분 못 하면 깊이 들어갈수록 헷갈림.

강의가 다루는 5개 개념

개념 1 Database / DBMS / Database System

제일 먼저 깨야 할 혼동. 일상에서 "DB" 라고 뭉뚱그려 부르지만 셋이 다름.

  • Database = 데이터 모음 자체 (파일·테이블 집합)
  • DBMS = 그 데이터를 관리하는 소프트웨어 (MySQL·Postgres·Mongo)
  • Database System = Database + DBMS + 응용 + 사용자 전체 시스템
왜 중요: "MySQL 이 느려" 라고 하면 *DBMS 문제* 임. "DB 가 커서 느려" 면 *Database 자체* 문제. 원인 진단할 때 어휘 맞아야 함.
개념 2 데이터 모델 (Data Models)

현실 세계를 데이터로 표현하는 *방식*. 같은 정보를 어떻게 구조화하느냐.

  • 관계형 모델 — 표(테이블)와 관계 (RDBMS)
  • 문서 모델 — JSON 같은 문서 (MongoDB)
  • 키-값 모델 — 키 1개로 값 조회 (Redis)
  • 그래프 모델 — 노드·엣지 (Neo4j)
핵심: 모델 선택 = 어떤 *질문에 빨라야 하는가* 결정. "이 사용자의 친구의 친구" → 그래프. "이 ID 의 데이터" → 키-값.
개념 3 스키마(Schema) vs 상태(State)

같은 DB 의 *틀* 과 *내용* 을 분리.

스키마 (Schema)
테이블 정의·컬럼·타입·제약. 거의 안 바뀜.
상태 (State)
지금 들어있는 실제 데이터 (행). 매 순간 바뀜.
실무 연결: 마이그레이션 = *스키마* 변경. INSERT/UPDATE = *상태* 변경. 두 개 다른 작업이라 운영 부담도 다름.
개념 4 3계층 아키텍처 (Three-schema Architecture)

"DB 변경했더니 앱이 다 깨짐" 막으려고 *추상화* 를 3층으로 쪼갠 설계.

  • 외부 스키마 (External) — 사용자·앱이 보는 뷰. 같은 DB 라도 사용자마다 다르게 보임
  • 개념 스키마 (Conceptual) — 전체 논리 구조. 진짜 테이블·관계 정의
  • 내부 스키마 (Internal) — 디스크 저장 방식. 인덱스·파일 구조
왜 중요: 내부(인덱스 변경) 바꿔도 개념(테이블) 안 바뀌고, 개념 바뀌어도 외부(뷰) 안 바뀌게 → *변경 격리*. 이게 좋은 시스템 설계의 핵심 패턴.
개념 5 DB 언어 — DDL / DML

DBMS 한테 명령 보내는 언어 종류. 역할로 나뉨.

DDL (Data Definition)
CREATE TABLE·ALTER·DROP. 스키마 다룸.
DML (Data Manipulation)
SELECT·INSERT·UPDATE·DELETE. 상태 다룸.
실무 팁: DDL = 보통 운영자/시니어. DML = 개발자 매일. DDL 실수 = 데이터 손실 위험 ↑ (그래서 마이그레이션 도구 따로).
